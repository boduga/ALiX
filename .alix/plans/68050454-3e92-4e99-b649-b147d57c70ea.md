## Summary
Add a `/healthz` endpoint to the application server to allow container orchestrators (like Kubernetes), load balancers, or monitoring tools to perform health checks.

## Changes
### Application Server
- **Action:** modify
- **File:** `src/index.ts` (or the primary entry point of the server, e.g., `src/app.ts` / `src/server.ts`)
- **Description:** Register a GET `/healthz` route. It should respond with a `200 OK` status and a JSON body containing basic diagnostic information:
  ```json
  {
    "status": "OK",
    "timestamp": "2023-10-27T12:00:00.000Z",
    "uptime": 123.45
  }
  ```

### Tests
- **Action:** create
- **File:** `src/__tests__/health.test.ts` (or equivalent test directory path)
- **Description:** Add integration tests to verify that hitting the GET `/healthz` endpoint returns a `200` status code and the correct JSON structure.

## Verification
- **Automated Tests:** Run the test suite (e.g., `npm test` or `jest`) to ensure the new health check tests pass successfully.
- **Manual Verification:** 
  1. Start the server locally (e.g., `npm run dev`).
  2. Send a request to the endpoint: `curl -i http://localhost:<port>/healthz`.
  3. Verify the response is `200 OK` with the expected JSON payload.

## Risk Assessment
- **Risk level:** low
- **Blast radius:** Minimal. This is a newly exposed route that does not modify any existing business logic or database interactions.
- **New dependencies:** None.
