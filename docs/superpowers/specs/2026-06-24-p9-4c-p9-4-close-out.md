## Core framing

P9.4 ships **3 of 5** `governance_change` mutation kinds. The 2 deferred kinds (`chain_restoration`, `governance_integrity`) are **investigation workflows**, not safe config mutations. They are reclassified to a future P9.6 `InvestigationRecommendation` / `OperatorTask` abstraction.

This is a documentation-only slice. No code or type changes. The runtime is already correct: `GovernanceChangeApplier` rejects both kinds with a clear unsupported-kind error, and proposals of those kinds cannot be applied.

## The mutation-capable invariant

A `governance_change` kind that participates in `GovernanceChangeApplier.apply()` MUST satisfy all four:

1. **Concrete target file** ŌĆö a specific file path the applier will read, validate, and write.
2. **Deterministic write semantics** ŌĆö given the same input payload + current file state, the mutation produces a known output (no agent calls, no external lookups).
3. **Drift guard** ŌĆö the payload carries a `currentXxx` value that the applier checks against the file's actual value, rejecting stale proposals.
4. **Revertable state** ŌĆö the file's previous state is captured by the existing `SnapshotStore` so `RevertApplier` can restore it.

Each of the 3 shipped kinds satisfies all four. The 2 deferred kinds do not.

## Why the 2 deferred kinds fail the invariant

**Investigation workflow** (as used below): a recommendation whose remediation requires human diagnosis, external pipeline changes, or operator action ŌĆö and which therefore has no deterministic on-disk target that `GovernanceChangeApplier` can mutate and revert.

### `chain_restoration`

Payload: `{ targetArtifactId, currentRate, targetRate }`. The drift finding's recommendation text is "Investigate why proposals are bypassing evidence chain provenance" ŌĆö that is an **investigation workflow**, not a config write.

| Invariant | Status | Why |
|-----------|--------|-----|
| Concrete target file | ŌØī | No config file controls evidence chain coverage. Chain usage is a measurement, not a knob. |
| Deterministic write | ŌØī | Restoring coverage requires operator action (e.g., fix the explain pipeline) ŌĆö outside the applier. |
| Drift guard | ŌØī | The `currentRate` is a measurement, not a stored value to check against. |
| Revertable state | ŌØī | There is no state to revert; the "fix" is a behavioral change in another subsystem. |

### `governance_integrity`

Payload: `{ issue, recommendationId }`. The recommendation text is "Investigate the review pipeline to determine why this rate is below threshold before any governance change" ŌĆö also an investigation.

| Invariant | Status | Why |
|-----------|--------|-----|
| Concrete target file | ŌØī | No config file controls integrity metric rates. |
| Deterministic write | ŌØī | Mutating an investigation acknowledgment is not a governance change. |
| Drift guard | ŌØī | Pure text + ID, no numeric field to check. |
| Revertable state | ŌØī | No state to revert. |

## Why we keep the kinds typed

`chain_restoration` and `governance_integrity` remain in the `Recommendation` and `GovernanceChangePayload` discriminated unions. Reasons:

1. **Removing them now risks breaking existing recommendations/proposals** that the recommendation generator has already emitted.
2. **The runtime already rejects them correctly** ŌĆö `GovernanceChangeApplier` throws a clear unsupported-kind error before any state change.
3. **The P9.5 dashboard will surface them** in a dedicated "investigation queue" panel, distinct from mutation-capable recommendations.
4. **The reclassification is non-breaking** ŌĆö P9.6 can add `InvestigationRecommendation` as a parallel type and migrate producers later.
